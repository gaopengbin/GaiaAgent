## 主要问题
- `step_update` 会串写到错误的 plan 卡片
- `reply` / `error` 到达顺序不稳定，导致 step 状态停留在 `pending` 或 `running`
- 聊天气泡、计划卡片、思考指示器在窄侧栏中会挤压头像
- Tauri 截图工具最终只显示 `screenshot executed`，没有显示内联图片

## 处理过程

### 1. 统一 Tauri 前后端事件绑定
- 为 Tauri 请求链路引入 `requestId`
- 将 `plan_generated`、`step_update`、`reply`、`error` 全部绑定到同一轮请求
- 避免使用单一的“最后一个 plan”指针造成事件误关联

### 2. 补齐最终回执
- 在 Rust `process_goal` 中补发 `reply` 与 `error`
- 空计划场景也返回最终消息
- 前端在 Tauri 模式下补齐对应监听，确保每次请求都有收尾消息

### 3. 修复聊天面板布局
- 调整 `ChatBubble.tsx`
- 调整 `PlanCard.tsx`
- 调整 `ThinkingIndicator.tsx`
- 改为基于剩余宽度自适应布局，解决头像被裁切和左右留白异常

### 4. 修复截图结果不内联显示
- 确认 Tauri 确实使用 `examples/web_ui/frontend` 作为前端根目录
- 确认 Cesium bridge 的 `screenshot` 真实返回中包含 `dataUrl`
- 在 `useBridgeWS.ts` 中把 `bridge.execute(...)` 的真实结果通过浏览器事件发出
- 在 `useTauriAgent.ts` 中优先消费 bridge 的真实结果，而不是只依赖 Rust `/api/command` 的投递确认
- 增加 `__lastBridgeResult` 兜底读取
- 修复一个更隐蔽的竞态：`reply` 先到、截图真实结果后到时，旧逻辑会提前清掉 step 关联，导致结果被丢弃
- 为此引入 pending step 关联缓存，等待 bridge 真实结果落地后再完成绑定

### 26. 数据资产追踪系统

**问题：** LLM 修改地图实体时猜测 ID（如 `marker_1`、`entity-0`），导致 `updateEntity` 报 "Entity not found"。

**根因分析：**
1. bridge 返回的 `addMarker` 结果含实际 `entityId`（GUID），但该结果通过 WS 异步到达
2. Rust `call_tool` 调用 `/api/command` 仅返回派发确认 `{ ok, sent, total }`，不含实际结果
3. `normalizeToolResult` 处理 HTTP 响应 → `{ output: "Command executed" }`，无 `meta`
4. `updateSceneState` 在 `onStepUpdate` 回调中调用时，bridge 结果尚未到达 → 使用回退 ID

**方案：**

*第一步 — 基础设施（types + normalizeToolResult + updateSceneState）：*
- `types.ts` — 新增 `ResultMeta { entityId?, layerId?, layerName? }`，`StepResult.meta?: ResultMeta`
- `executor.ts` — `normalizeToolResult` 递归提取 `entityId`/`id`/`name` 到 `meta`
- `agent/types.ts` — `SceneLabel` 新增 `id: string` 字段
- `history.ts` — `updateSceneState` 使用 `step.result?.meta` 获取真实 ID；`formatSceneContext` 输出 `id="xxx"` 供 LLM 引用；补齐 `clearAll`/`removeEntity` 分支

### 33. load3dTiles Ion 支持 + Bridge 自动更新机制

**问题：** `load3dTiles` 加载 Cesium Ion 资产报 `Cannot read properties of undefined (reading 'extension')`。

**根因：** bridge 的 `load3dTiles` 只解构 `url` 参数，runtime 传递的 `ionAssetId` 被忽略 → `Cesium3DTileset.fromUrl(undefined)` → 崩溃。npm 发布的 bridge@1.139.15 已修复此问题（含 `fromIonAssetId` 分支），但项目 `public/` 下的 bridge 文件是旧版本（3月30日构建）。

**修复：**
1. **Bridge 自动更新机制** — 以 `cesium-mcp-bridge` 作为 devDependency，`predev`/`prebuild` 钩子自动从 `node_modules` 复制最新 bridge 到 `public/`
   - `scripts/update-bridge.mjs` — 复制脚本
   - `package.json` — 新增 `update-bridge`/`predev`/`prebuild` 脚本 + devDependency
2. **prompt 更新** — 告知 LLM `load3dTiles` 支持 `assetId` 参数加载 Ion 资产
3. tauri-app 和 web_ui/frontend 两个项目均同步实现
